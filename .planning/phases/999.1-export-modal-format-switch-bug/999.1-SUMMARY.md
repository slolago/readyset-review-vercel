# Backlog 999.1: ExportModal format-switch bug — SHIPPED

**Shipped:** 2026-04-22 (inline, outside the normal phase flow)

## The bug (as reported)

Open Export modal on an asset → export as GIF → don't close modal → switch
format toggle to MP4 → click Export again → second "export" produces the
GIF again (and mirror: MP4 → GIF also stuck on MP4).

## Actual root cause (diagnosis differed from backlog note)

Original backlog hypothesis was "stale closure on `format` in `handleSubmit`'s
`useCallback` deps." That was wrong — `format` IS in the deps array at
line 216, so the closure captures fresh values.

The real issue was in the footer's button-swap logic:

```tsx
{ui === 'ready' && signedUrl ? (
  <button onClick={handleDownload}>Download</button>
) : (
  <button onClick={handleSubmit}>Export</button>
)}
```

After a successful export, `ui === 'ready'` and `signedUrl` is set, so the
button shows **Download** — not Export. When the user toggled format to
MP4 and clicked the (still-labeled-Download) button, `handleDownload` fired:

```tsx
forceDownload(signedUrl, `${filename}.${format}`);
```

`signedUrl` was the GCS URL of the already-encoded GIF, but `format` was
now `'mp4'`. The browser downloaded the GIF with a `.mp4` filename — looks
like an MP4 download from the browser's history, opens as a GIF when the
user tries to play it. From the user's perspective: "it exports a GIF
even though I chose MP4."

## The fix (one `useEffect`)

```tsx
useEffect(() => {
  if (!open) return;
  if (ui === 'ready' || ui === 'failed') {
    setUi('idle');
    setSignedUrl(null);
    setErrorMsg(null);
  }
}, [format, inPt, outPt, filename]);
```

Any change to the export params (format / in / out / filename) after a
successful or failed encode now invalidates the cached result — the footer
button swaps back from Download to Export, so clicking does a fresh encode
with the new params.
