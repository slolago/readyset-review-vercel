<objective>
Restore project-scoped review-link listing broken since Phase 44's access-model rewrite.

Purpose: `GET /api/review-links?projectId=X` returns 500 in production because the query `.where('projectId','==',X).orderBy('createdAt','desc')` requires a Firestore composite index that was never deployed. Every project-scoped caller (Add-to-Link modal, Project Review Links tab, sidebar shortcut page) catches the 500 and renders either an error toast or an empty state. The sibling endpoint `/api/review-links/all` already solves this by fetching without `orderBy` and sorting in memory — we apply the same pattern here. Three clients all send the correct query shape already (`?projectId=X` + `Authorization: Bearer`), so the fix is server-only plus a regression test.

Output:
- `src/app/api/review-links/route.ts` — list endpoint no longer requires composite index
- Client call sites audited; no behavior change required (they already match the contract)
- Client empty-state copy tightened so "zero links" doesn't masquerade as an error
- Integration tests in `tests/permissions-api.test.ts` covering GET /api/review-links across roles
</objective>

<audit>
Server contract (src/app/api/review-links/route.ts GET, current HEAD):
  - Auth: Authorization: Bearer <idToken> (getAuthenticatedUser)
  - Query: ?projectId=<id>  (400 if missing)
  - Gate: canAccessProject(user, project)  (403 if not)
  - Query: db.collection('reviewLinks')
             .where('projectId','==', projectId)
             .orderBy('createdAt','desc')
             .get()
  - Response: { links: [{ id, ...safe, hasPassword }] }

Client callers (all three already send the correct shape):
  1. src/components/review/AddToReviewLinkModal.tsx (L57-77)
       fetch(`/api/review-links?projectId=${projectId}`, { headers: { Authorization: `Bearer ${token}` } })
  2. src/components/review/ReviewLinksTab.tsx (L24-38)
       same shape
  3. src/app/(app)/projects/[projectId]/review-links/page.tsx (L56-74)
       same shape

Global page uses a different endpoint:
  src/app/(app)/review-links/page.tsx → GET /api/review-links/all  (unaffected by this bug)

ROOT CAUSE:
  `.where('projectId','==',X).orderBy('createdAt','desc')` requires a composite
  index on reviewLinks(projectId ASC, createdAt DESC). That index is not
  deployed. Firestore throws FAILED_PRECONDITION → bare `catch {}` at L44
  returns 500 `{ error: 'Failed to fetch review links' }`. All three clients
  treat this as "Failed to load review links" and bail.

  Evidence that `/all` already dodged this trap intentionally:
    src/app/api/review-links/all/route.ts:31
    // no orderBy — avoids composite index requirement; sorted in-memory below

FIX:
  Mirror `/all`'s pattern in the project-scoped list: drop the `.orderBy(...)`
  from the Firestore query; sort the resulting array in JS by `createdAt` desc
  using the same `_seconds`-aware comparator. No new index required; no client
  changes; no permission changes.
</audit>

Two small correctness tweaks so "zero links" reads as empty, not as an error:

1. `src/components/review/ReviewLinksTab.tsx` — the current `fetchLinks` does `const data = await res.json(); setLinks(data.links ?? []);` unconditionally. If the server returns 500 (before this phase's fix) or 403, `res.json()` may still parse and `data.links` is undefined → silently empty state with no toast. Add a `if (!res.ok) throw new Error('Failed to load review links');` before the json parse so real failures surface via the existing `catch` toast, while 200 + empty array renders the existing "No review links yet" state cleanly.

2. `src/app/(app)/projects/[projectId]/review-links/page.tsx` (L56-74) — same issue: `const data = await res.json(); setLinks(data.links ?? []);` inside a try/catch that "silently fails". Add `if (!res.ok) throw new Error('Failed to load');` before json parse so the finally block still runs and loading ends, but we don't render "No review links yet" when the server actually errored. The catch can stay silent-ish (the comment says "silently fail — empty state will show"); change it to log the error to console so production issues are observable: `console.error('review-links list fetch failed:', err);`.

3. `src/components/review/AddToReviewLinkModal.tsx` — already does `if (res.ok) { ... } else { toast.error('Failed to load review links'); }`. No change needed.

Do NOT redesign these components. Do NOT change auth token refresh logic. Do NOT add loading skeletons or new UI. Just the two `if (!res.ok) throw` guards.
