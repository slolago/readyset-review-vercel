**Scout the client first (5 min):** `src/app/(app)/admin/page.tsx` stores `users` and `projects` as flat `useState` arrays fed from one-shot `data.users` / `data.projects` reads. There is no pagination UI, no infinite-scroll sentinel, and the Projects tab is already eager-fetched in parallel with Users (Phase 75 PERF-17).

**Decision (documented):** Adding full "Load more" pagination UI to the admin tables is too invasive for one surgical task (would require infinite-scroll state + skeleton + ProjectsTable/UserTable row-append logic). Instead, ship **option (b)** from the context: API-side bounded pagination with a larger default page size, and have the admin client request the first page with the default limit. This still kills unbounded scans and caps OOM risk; full pagination UI is deferred to a future plan.

**API changes (all 3 routes):**
1. Accept `?limit=N` (default 50, clamped to `[1, 100]`) and `?cursor=<docId>` query params.
2. Build the query with `.orderBy('createdAt', 'desc').limit(limit)`. If `cursor` is provided, fetch that doc and apply `.startAfter(cursorDoc)` before `.limit()`.
3. Response shape: `{ users | projects | links: [...], nextCursor: string | null }`. `nextCursor` = `snap.docs[snap.docs.length - 1].id` iff `snap.size === limit`, else `null`.
4. `/api/admin/users/route.ts:10-18` ŌĆö replace the single `.get()` with the paginated pattern above. Keep the existing `.orderBy('createdAt', 'desc')`.
5. `/api/admin/projects/route.ts:13` ŌĆö same: `.orderBy('createdAt', 'desc').limit(limit)` with optional `startAfter`. Keep the owner-batch enrichment (`db.getAll` + `ownerMap`) unchanged ŌĆö it already runs against the paginated slice.
6. `/api/review-links/all/route.ts` ŌĆö this route's pagination unit is review links, not projects. Approach: keep the existing per-project filter + `in`-chunk scan that produces `allLinks`, then apply **in-memory slicing** via `cursor` (last link's id) + `limit` after sorting by `createdAt` desc. The user-project list is bounded by project access, and review links per user are the scale concern. Response: `{ links: [...], nextCursor }`. Comment count queries stay scoped to the sliced page's tokens only (one chunked `in` query max). Document in SUMMARY that a true cursor-per-reviewLink with Firestore-side ordering would require a new composite index + cross-project union and is deferred.

**Client change (minimal ŌĆö admin page):** `src/app/(app)/admin/page.tsx` still sets `setUsers(data.users)` and `setProjects(data.projects)` ŌĆö no UI change needed. The client simply gets a bounded first page (default 50) instead of everything. Note inline in a comment at each fetch call: `// Bounded by API ŌĆö pagination UI deferred (see 78-01 PLAN).`
