  <name>Task 6 (VWR-06): VersionComparison duration effects re-subscribe on version swap</name>
  <files>src/components/viewer/VersionComparison.tsx</files>
  <action>
  In src/components/viewer/VersionComparison.tsx at ~line 343-358, the two duration effects currently have `[]` deps. When the user swaps to a different version on side A or B, the video element's `src` changes but the listener is still attached â€” `loadedmetadata` fires against a new currentTime-ready event that the effect no longer reacts to (the listener is stale because we're accessing `videoARef.current` captured at mount time, which is still the SAME video element instance â€” so this specific bug may actually be about `durationA` not updating because we already read it at mount via `readyState >= 1` but don't re-read after a src flip).

  The root cause: when `selectedIdA` changes, the <video> element persists (same ref) but its `src` prop re-renders to `urlA`. The browser fires `loadstart` â†’ `loadedmetadata` on the new src. The existing listener IS still attached so this would normally work â€” but the cleanup runs only on unmount (`[]` deps). That's actually fine for the listener. The real bug is that between the src change and the new `loadedmetadata` firing, `durationA` holds the OLD value, and there's no reset.

  The fix (simplest, matches the already-present pattern on dimsA/dimsB at line 281-282):

  1. Replace `[]` with `[selectedIdA]` and `[selectedIdB]` on the two duration effects. When the selection changes, the cleanup runs (detaches old listener), then re-attaches a fresh listener and re-reads duration if `readyState >= 1`:

  ```ts
  useEffect(() => {
    const vA = videoARef.current;
    if (!vA) return;
    const onLoaded = () => setDurationA(vA.duration || 0);
    vA.addEventListener('loadedmetadata', onLoaded);
    // If metadata already arrived (cached), read now; otherwise wait for event.
    if (vA.readyState >= 1) setDurationA(vA.duration || 0);
    else setDurationA(0); // reset so UI doesn't show stale duration while the new src loads
    return () => vA.removeEventListener('loadedmetadata', onLoaded);
  }, [selectedIdA]);

  useEffect(() => {
    const vB = videoBRef.current;
    if (!vB) return;
    const onLoaded = () => setDurationB(vB.duration || 0);
    vB.addEventListener('loadedmetadata', onLoaded);
    if (vB.readyState >= 1) setDurationB(vB.duration || 0);
    else setDurationB(0);
    return () => vB.removeEventListener('loadedmetadata', onLoaded);
  }, [selectedIdB]);
  ```

  2. Verify the cleanup runs BEFORE the new effect body (React guarantees this) so the old listener is detached before the new one attaches. Since it's the same video element, a duplicate listener would be harmless (same fn identity â†’ dedup) but correct lifecycle is cleaner.

  3. Do NOT switch to an inline `onLoadedMetadata` handler on the <video> â€” the existing pattern matches VideoPlayer and is fine once the deps are corrected.

  4. Do NOT touch the dimsA/dimsB or errA/errB effects (already have correct deps at line 281-282).

  Disable the exhaustive-deps lint line if needed, but the deps here are legitimately complete.
  </action>
