### Phase 49: metadata-accuracy
**Goal**: Info panel shows accurate, type-appropriate metadata for every asset — videos keep the ffprobe-derived field set; images show real width/height/size/EXIF and do not surface video-only fields. The upload date always renders as a readable date.
**Depends on**: Nothing (first phase of v1.8)
**Requirements**: MEA-01, MEA-02, MEA-03, MEA-04
**Success Criteria** (what must be TRUE):
  1. Uploading a JPEG with real dimensions 2250×4000 / 718 KB shows the Info panel as 2250×4000 / 718 KB — no downscaling, no truncation, no transcoded/thumbnail values
  2. The Info panel for an image asset does not show Container, Pixel format, Color space, or Overall bitrate fields; it does show image-relevant fields (resolution, size, optional EXIF/color-profile)
  3. The Info panel for a video asset continues to show the current v1.6 field set (container, codecs, bitrate, color, frame rate, audio)
  4. The Upload → Date row renders a human-readable date (e.g. "Apr 20, 2026, 3:42 PM") for assets uploaded before and after this phase; never "Invalid Date"

### Phase 50: review-links-repair
**Goal**: Review-link discovery works from every entry point — the "Add to review link" modal shows existing links for the asset's project, and the project-scoped Review Links views list the project's links.
**Depends on**: Phase 49
**Requirements**: RVL-01, RVL-02
**Success Criteria** (what must be TRUE):
  1. Opening "Add to review link" on an asset shows the project's existing review links with no error toast
  2. The sidebar's Review Links shortcut under a project lists every review link that has that project's `projectId`
  3. The Review Links tab inside a project's page lists the same set of links as the sidebar shortcut — no empty state when links exist
  4. The global Review Links page (if separate) continues to work and shows all links the user has access to

**Depends on**: Phase 49
**Requirements**: UPL-01, UPL-02, UPL-03, UPL-04

### Phase 52: trash-and-recovery
**Goal**: Deleting an asset or folder never destroys data immediately — users have a Trash view to restore or permanently delete items.
**Depends on**: Phase 50
**Requirements**: TRH-01, TRH-02, TRH-03
**Success Criteria** (what must be TRUE):
  1. Deleting an asset or folder removes it from all normal views but keeps it in Firestore with a `deletedAt` timestamp
  2. A Trash view (project-level) lists every soft-deleted asset/folder for that project with name, deleted date, and a Restore action
  3. Restoring returns the item to its original folder — or to project root when the original folder is also deleted — and clears `deletedAt`
  4. From Trash, the user can permanently delete a single item or "Empty Trash" to permanently delete all; permanent deletion frees the GCS object and removes the Firestore doc
  5. Permanent delete confirmation uses the in-app `useConfirm` dialog with a distinct "This cannot be undone" warning
**UI hint**: yes
